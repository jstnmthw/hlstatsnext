# GraphQL Audit: all

**Date:** 2026-05-24
**Scope:** `apps/api/src/` (Pothos schema, Yoga server, modules, context, exclusion lists), `apps/web/src/` GraphQL operations + Apollo wiring, `packages/db/prisma/schema.prisma` (source of truth for field exposure).
**Estimated security:** **High**

## Summary

5 critical, 7 warning, 8 info findings. The exclusion-list system is mostly sound for `User`, `Session`, `Account`, `Verification`, `ServerToken`, and `EventRcon` _outputs_, but two heavy-PII models (`Player`, `EventConnect`) were never classified and are auto-generated as public — leaking email, real names, IP addresses, and geo-coordinates to unauthenticated callers. Separately, auto-generated `*WhereInput` types still filter on sensitive fields (`Player.email`, `ServerToken.tokenHash`, `Server.rconPassword`, `EventRcon.password`), enabling enumeration / binary-search attacks even when output objects strip those fields. The web codegen artefact is stale and still types `Server.rconPassword`, so an admin page selects it (`apps/web/src/features/admin/servers/graphql/server-queries.ts:60`) and ships it to the browser if the API ever re-exposes it.

## Findings

### [CRITICAL] Player PII (email, IP, real name, geo) auto-exposed to unauthenticated clients

- [x] **File:** `apps/api/src/pothos-schema.ts:21-29` (model not classified) + `packages/db/prisma/schema.prisma:94-115` (sensitive columns) + auto-generated Pothos object via `generateAllObjects` at `pothos-schema.ts:69`
      **Category:** exclusion-list
      **Description:** `Player` appears in _none_ of the four exclusion lists, so the Pothos Prisma plugin auto-generates a public `Player` object exposing every column — including `email`, `lastAddress` (last known IP), `fullName`, `lat`, `lng`. The auto-generated `findManyPlayer`/`findUniquePlayer` queries are unauth-public. Any caller can run `query { findManyPlayer { email lastAddress fullName lat lng } }` and dump all PII from the player table.
      **Evidence:**
      `prisma
// packages/db/prisma/schema.prisma:100-115
lastAddress String  @default("") @map("last_address") @db.VarChar(32) // Last known IP address
fullName    String? @map("full_name") @db.VarChar(128) // Optional full/real name
email       String? @map("email") @db.VarChar(64) // Optional email
...
lat     Float?  @db.Float
lng     Float?  @db.Float
`
      `ts
// apps/api/src/pothos-schema.ts:21-29 — Player absent
const AUTH_MODELS = ["Session", "Account", "Verification"] as const
const CUSTOM_MODELS = ["ServerToken"] as const
const SENSITIVE_MODELS = ["Server", "EventRcon"] as const
const ADMIN_ONLY_MODELS = ["User"] as const
`
      **Remediation:** Add `Player` to `SENSITIVE_MODELS` and ship `apps/api/src/modules/player/player.object.ts` defining a custom `builder.prismaObject("Player", …)` that omits `email`, `lastAddress`, `fullName`, `lat`, `lng` (or admin-gates them via field-level `requireAdmin` resolvers). Mirror the existing `server.object.ts` pattern.

### [CRITICAL] `PlayerWhereInput` allows unauth enumeration of email / IP / real name

- [x] **File:** auto-generated by `@repo/db/graphql/types` from `schema.prisma`, surfaced in codegen at `apps/web/src/lib/gql/graphql.ts:20690,20695,20704,20708,20710` (and exposed on `findManyPlayer` / every `Player`-relating WhereInput)
      **Category:** sensitive-exposure
      **Description:** Even if `Player` outputs are eventually stripped, the auto-generated `PlayerWhereInput` exposes filters on `email: StringNullableFilter`, `fullName: StringNullableFilter`, `lastAddress: StringFilter`, `lat: FloatNullableFilter`, `lng: FloatNullableFilter`. An unauthenticated client can binary-search emails/IPs character-by-character: `findManyPlayer(where: { email: { startsWith: "a" } }) { playerId }` returns a count — repeat to recover full strings, or check `email: { equals: "victim@example.com" }` to confirm a specific email is in the DB and which `playerId` it maps to.
      **Evidence:**
      `ts
// apps/web/src/lib/gql/graphql.ts — PlayerWhereInput excerpt
email?: InputMaybe<StringNullableFilter>;
fullName?: InputMaybe<StringNullableFilter>;
lastAddress?: InputMaybe<StringFilter>;
lat?: InputMaybe<FloatNullableFilter>;
lng?: InputMaybe<FloatNullableFilter>;
`
      **Remediation:** The Prisma plugin generates inputs based on the datamodel, not the custom object — you must either (a) post-process the generated input types in `@repo/db/graphql/types` codegen to strip sensitive columns, (b) substitute a custom `PlayerWhereInput` that whitelists allowed filters, or (c) move all `Player` queries behind a sanitiser resolver. Option (b) is the cleanest: register custom queries `findManyPlayer` / `findUniquePlayer` / `countPlayer` with a hand-written input type, and exclude `Player` from `generateAllQueries`.

### [CRITICAL] `ServerToken.tokenHash` filterable via `ServerTokenWhereInput` — enables offline token cracking

- [x] **File:** auto-generated; surfaced in codegen at `apps/web/src/lib/gql/graphql.ts:26760-26779` (`ServerTokenWhereInput`); reachable via `Server.authToken` chain because `Server` is in `SENSITIVE_MODELS` (object stripped, queries public) and `Server.authToken` relation is exposed at `apps/api/src/modules/server/server.object.ts:61`.
      **Category:** sensitive-exposure
      **Description:** `ServerToken.tokenHash` is a deterministic SHA-256 (`@unique @map("token_hash") @db.VarChar(64)` at `packages/db/prisma/schema.prisma:758`). `ServerTokenWhereInput.tokenHash: StringFilter` is auto-generated, and the input is reachable as `ServerWhereInput.authToken: ServerTokenWhereInput`. An unauthenticated client can binary-search the 64-hex-char hash via `findManyServer(where: { authToken: { tokenHash: { startsWith: "ab" } } }) { serverId }` and recover the hash of every active token. With the hash, an offline dictionary / brute-force attack against the raw token becomes possible (`tokenHash = sha256(rawToken)` per `packages/crypto`). A recovered raw token authenticates the daemon to ingest fabricated events for that server.
      **Evidence:**
      `ts
// apps/web/src/lib/gql/graphql.ts:26771,26774
rconPassword?: InputMaybe<StringFilter>;
...
tokenHash?: InputMaybe<StringFilter>;
`
      `ts
// apps/api/src/modules/server/server.object.ts:61
authToken: t.relation("authToken", { nullable: true }),
`
      **Remediation:** Same as the Player WhereInput fix — strip credential / hash columns from auto-generated `*WhereInput` for `ServerToken`, `Server`, `EventRcon`. Either patch the input-type codegen in `@repo/db/graphql/types` to exclude fields by-name (`/password|secret|token|hash/i`), or wrap the relevant queries (`findManyServer`, `Server.authToken`) with custom whitelisting inputs. Additionally consider tightening: tokens are admin credentials, and exposing them via the `Server` relation at all (even hashes) is risky — consider gating `Server.authToken` behind `requireAdmin` at the relation level.

### [WARNING] `Server.rconPassword` / `EventRcon.password` filterable via WhereInput

- [x] **File:** `apps/web/src/lib/gql/graphql.ts:28544` (`ServerWhereInput.rconPassword`), `:7433,7448` (`EventRconWhereInput.password`)
      **Category:** sensitive-exposure
      **Description:** Same class as the tokenHash issue, but the encryption properties limit exploitability: `Server.rconPassword` is AES-256-GCM ciphertext (per the `// AES-256-GCM encrypted` comment at `schema.prisma:761`) with a per-encryption random IV, so the same plaintext encrypts to a different ciphertext each time — binary search of the ciphertext bytes does not recover plaintext. However, the filter still allows existence-checking specific ciphertext blobs and timing attacks. `EventRcon.password` is `VarChar(128)` with no `// encrypted` annotation in `schema.prisma:493` — it is the raw RCON command password as parsed from server log lines, almost certainly stored as plaintext. If so, binary search via `findManyEventRcon(where: { password: { startsWith: "..." } }) { id }` recovers actual RCON passwords character-by-character.
      **Evidence:**
      `prisma
// packages/db/prisma/schema.prisma:486-494
model EventRcon {
  id        Int       @id @default(autoincrement())
  ...
  password  String    @default("") @db.VarChar(128)  // ← no encryption annotation
  ...
}
`
      **Remediation:** Strip these fields from the auto-generated WhereInputs (same mechanism as the tokenHash fix). Audit `apps/daemon/src/modules/rcon/` to confirm whether `EventRcon.password` is ever written with a non-empty value (if always empty, the field is dormant but should still be omitted from inputs).

### [WARNING] Yoga CORS origin defaults to `undefined` if `FRONTEND_URL` is unset in production

cors: {
  origin: isProduction ? process.env.FRONTEND_URL : "http://localhost:3000",
  credentials: true,
},
`
      **Remediation:** Add a startup guard: `if (isProduction && !process.env.FRONTEND_URL) throw new Error("FRONTEND_URL required in production")`. Optionally validate via Zod alongside `ENCRYPTION_KEY`.

## Stable patterns found

- `AUTH_MODELS` exclusion (`Session`, `Account`, `Verification`) is correctly applied to both objects and queries — no Better Auth internals reachable.
- `User` admin gating via `requireAdminResolver` wraps every auto-generated User query and every auto-generated mutation; the wrapper correctly distinguishes `prismaField` vs plain `field` resolver signatures.
- Custom `ServerToken` object exposes a `hasRconPassword` boolean instead of the ciphertext — good pattern, mimic elsewhere.
- All Web server actions (`apps/web/src/features/admin/{servers,tokens,users}/actions/*.ts`) check `getSession()` + permission before mutating — defence in depth on top of the GraphQL-level admin gate.
- `(admin)/layout.tsx` redirects non-admins, providing route-level gating in addition to per-operation auth.
- Pothos `Prisma.$executeRaw` calls in `apps/api/src/modules/server/server.repository.ts` are all tagged-template parameterised — no `$executeRawUnsafe`, no SQL injection.
- No `.map(async ...)` / `forEach(await ...)` N+1 patterns in `apps/api/src`.
- Yoga `maxDepthPlugin(7)` unconditionally enabled; `useDisableIntrospection()` enabled in production; `maskedErrors: isProduction` enabled.
- All Apollo operations in `apps/web` use the typed codegen helper from `@/lib/gql` — no hand-rolled `gql\`\`` strings.
- Apollo `getClient` correctly forwards browser cookies into server-side fetches for SSR (`apps/web/src/lib/apollo-client.ts:17-32`).

## Recommendations

**Quick wins (1 PR, can ship today):**

1. Add `Player` and `EventConnect` to `SENSITIVE_MODELS`; write `player.object.ts` and `event-connect.object.ts` mirroring `server.object.ts`. Removes the dominant PII leak.
2. Drop `email` from `GET_PLAYERS_QUERY` / `GET_PLAYERS_WITH_PAGINATION` in `apps/web/src/features/admin/players/graphql/player-queries.ts`; create a separate admin-only `GET_ADMIN_PLAYERS_QUERY` that keeps email and is only imported under `app/(admin)/`.
3. Add startup guard for `FRONTEND_URL` in prod (`apps/api/src/index.ts`).
4. Re-run `pnpm codegen` and `pnpm check-types`; fix all fallout (`rconPassword` in admin server edit, dead `findManyAccount` etc).
